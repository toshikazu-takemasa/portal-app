# ADR-001: GitHub Contents API をプライマリストレージとして使用

日付: 2026-04-07  
状態: 採用

## 背景

ローカルPCなしで動作するクラウド完結設計が必要。バックエンドサーバーを持たない静的HTMLアプリとして動作させたい。データの永続化・バージョン管理・マルチデバイス同期が要件。

## 検討した選択肢

1. **GitHub Contents API**（PUT/GET でMarkdownファイルを読み書き）
2. **localStorage のみ**（ブラウザ内完結、同期なし）
3. **Cloudflare D1**（SQLite、Cloudflare Workers が必要）
4. **Firebase / Supabase**（外部サービス依存）

## 決定

GitHub Contents API を採用。PAT（Personal Access Token）で認証し、リポジトリのファイルを直接読み書きする。

## 理由

- 既存コード（my-portal, docs）が既にこのパターンを使用しており、移行コストが最小
- バックエンドサーバー不要で静的HTMLのまま動作
- Markdownファイルとして保存することでGitの履歴・差分・バックアップが自動化される
- PAT のスコープをリポジトリ単位で制限できるため、仕事/個人の分離が可能

## トレードオフ・リスク

- 全文検索・集計クエリが難しい（ファイル一覧取得 + クライアント側フィルタリングになる）
- API レートリミット（認証済みで5000回/時間）
- PAT がブラウザ localStorage に保存されるためXSSリスクがある

## 将来の再評価タイミング

音声機能追加時。音声→テキスト変換結果の高速検索・集計が必要になった場合に Cloudflare D1 への移行を検討する（→ ADR-003）。
