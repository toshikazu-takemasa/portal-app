# ADR-027: 日記アプリのキャッシュ更新機能の追加

日付: 2026-05-01
状態: 提案

## 背景

現在、Portalアプリの他のモジュール（タスク、家計管理など）ではキャッシュの更新や再取得の仕組みが導入されているが、日記（Journal）アプリにおいてはキャッシュ更新機能が不足しているという課題が GitHub Issue #2 で指摘されている。
別端末で日記を編集した場合や、リモートリポジトリ側で直接Markdownファイルが変更された場合、ローカルのキャッシュ（またはブラウザ上の古い状態）が優先されてしまい、最新のデータがUIに反映されない可能性がある。

## 決定

**日記アプリ（`pages/journal.tsx`等）において、リモート（GitHub等）から最新の日記データを明示的または自動的にフェッチし、キャッシュを更新する機能（Refresh）を実装する。**

### アーキテクチャ

1. **手動更新 (Manual Refresh)**
   - 日記のUIヘッダー等に「更新」ボタンを配置し、ユーザーの操作によってAPI（例: `/api/github/file?path=vault/journal/...`）から最新のMarkdownファイルを取得する。
   - 取得後、エディタの状態（ステート）を新しい内容で上書きする。
2. **自動検知・更新 (Auto Refresh / Validation)**
   - マウント時、あるいはタブがアクティブになった時（Focus時）にバックグラウンドでリモートのファイルのハッシュや最終更新日時をチェックする（SWRなどのライブラリを活用する）。
   - ローカルの編集中データが存在する場合の競合（コンフリクト）を避けるため、リモート側が新しい場合は「新しいバージョンがあります」という通知を出し、ユーザーの許可を得てから再読み込みを行う（あるいは差分マージを行う）よう考慮する。

## トレードオフ

- **メリット:** 複数端末でシームレスに日記を編集できるようになり、古いデータで上書きしてしまう（デグレ）事故を防ぐことができる。
- **デメリット:** 編集中に裏でキャッシュ更新が走ると、ユーザーが書いている最中のテキストが消失するリスクがある。
  - *対策:* ADR-018 で導入した「日記の自動保存（ドラフト保存）」機能と統合し、ドラフトとリモート版の差分をユーザーが選択できるUI設計が必須となる。

## 次のステップ

- 日記ページへの手動リロード（再フェッチ）ボタンのUI追加。
- データの競合解決（ローカルのドラフトとリモートの最新版の扱い）についての詳細設計。
