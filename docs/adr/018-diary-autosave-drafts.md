# ADR-018: 日記の入力中ドラフト（下書き）の自動保存

日付: 2026-04-21
状態: 提案（検討中）

## 背景

現在、日記ページにおいて、日記を書きかけている途中で別の画面（例：チャット画面やタスク画面など）へ遷移してしまうと、それまで記述していた内容が破棄されてしまう問題があります。ユーザーが誤ってページを移動した場合や、リファレンスのために別の情報を確認しようと画面を切り替えた場合に、書きかけのコンテキストが失われてしまいます。

この不便さを解消するため、手軽に入力中の状態を保存しておける仕組みを導入する必要があります。

## 検討の方向性（主なアプローチ）

入力された文字列を失わないための手軽な仕組みとして、以下の3つのアプローチを検討します。

ユーザー回答：1を選択
### 1. フロントエンドのストレージ (`localStorage` 等) を使う方法
Reactコンポーネント上で入力が変更されるたびに（あるいは一定時間入力が止まったタイミングで）、ブラウザの `localStorage` または `sessionStorage` にテキスト内容を保存する。
画面を再度開いた際に、保存された下書きがあれば自動で復元する、もしくはユーザーに復元するかどうかを尋ねる。
*   **メリット:** 実装が非常にシンプルで手軽。バックエンドへの通信やファイルI/Oが発生しない。
*   **デメリット:** 異なるデバイスやブラウザ間でのドラフトの共有はできない。

### 2. グローバルステート管理 (Zustand `persist` 等) を使う方法
Zustand などの状態管理ライブラリで入力中の文字列を管理し、`persist` ミドルウェアを用いて永続化する。
*   **メリット:** Zustandがすでに導入されていれば、設定追加のみで比較的スマートに実装できる。
*   **デメリット:** アプローチ1と同様、ブラウザ内にデータが留まる。

### 3. Vault（ファイルストレージ）へのバックグラウンド保存
数秒間の遅延等（デバウンス処理）を利用して、バックエンドに対して下書き用の一時ファイル（例: `.draft.md` や SQLiteの特定のテーブル）に状態を保存する。
*   **メリット:** 別のPCやスマートフォンからアクセスした際にも下書きが共有される。
*   **デメリット:** ネットワークリクエストとI/Oが頻発するため、実装コストやサーバーへの負荷が高くなる。

## 今後の決定に向けて

最も手軽かつ要件を満たせるのは、**「1. `localStorage` を利用したクライアントサイドでの自動保存」** だと考えられます。
複雑なデバイス間の同期が不要であれば、この仕組みをベースに「日記入力フォーム」のコンポーネントを修正する方針を推奨します。
