# ADR-017: 家計管理におけるデータベース移行とAPI連携レイヤーの導入

日付: 2026-04-16
状態: 採用

## 背景

現在、家計録データは `vault/finance/YYYY-MM.json` 形式で月ごとに管理・保存されています。しかし、データ量の増加や、集計クエリの複雑化を考慮すると、ファイルシステム・JSONベースの管理ではパフォーマンス・拡張性の面で限界があります。
また、将来的な構想として、ZaimやMoneyForwardといった個人向けの家計簿アプリ（銀行やクレジットカードの自動連携機能を持つシステム）からAPIを経由してポータルにデータを統合することが求められています。

## 決定

**家計録データをJSONから SQLite に移行し、外部APIからデータを取り込むための連携レイヤーを新設する。**

### 1. SQLiteの採用とテーブル設計
- D1等への完全移行はADR-003で保留となっているため、まずはローカル環境で扱いやすい **SQLite**（`better-sqlite3` と `Drizzle ORM` などの組み合わせ、または生クエリでの初期対応）を採用する。
- 基本的なテーブル構造として、以下を定義する。
  - `transactions`: 取引履歴（金額、日付、説明、外部参照IDなど）
  - `accounts`: 対象の口座情報
  - `categories`: 支出・収入の分類カテゴリ

### 2. API連携レイヤー（Finance Integration Provider）
- Taskドメインで導入した Architecture (BacklogやGitHub等の `IntegrationProvider`) を Finance ドメインにも適用する。
- インターフェースとして `FinanceIntegrationProvider` を定義し、各種外部サービスとの同期ロジックをプラグイン形式で追加できるようにする。
- データの流れ: `外部API` → `API Route Proxy` → `FinanceIntegrationProvider` → `SQLite`

## トレードオフ

- **メリット:** SQLによる柔軟な集計・検索クエリが記述可能になり、将来的なBI化やダッシュボード表示の基盤が整う。APIレイヤーの標準化により外部サービスへの依存を分離できる。
- **デメリット:** SQLiteファイルの管理（共有やバックアップなど）が新たに必要になる。既存のVaultベースの運用（Git管理）とはデータフローが異なるため、SQLiteデータベースファイル自体をポータルのバックアップ対象としてどのように扱うか運用設計が必要。
