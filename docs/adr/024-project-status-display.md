# ADR-024: 課題アプリにおけるプロジェクト情報（プロジェクト名・システムステータス）の表示

日付: 2026-04-26  
状態: 提案

## 背景

現状の Backlog アプリ（課題アプリ）は、複数のプロジェクトから自分にアサインされた課題を一覧表示している。しかし、現在の `UnifiedTask` 型および UI では以下の情報が不足している：

1. **どのプロジェクトの課題か直感的に分からない**: 課題キー（SPR-123等）のみが表示され、プロジェクト名が表示されていない。
2. **プロジェクト固有のステータス名が失われる**: Backlog のカスタムステータス（例：「レビュー待ち」「保留」）が、汎用的な「未対応」「処理中」等にマッピングされる際に詳細が失われる。

ユーザーより「プロジェクトのステータスがあればそれも表示するようにしたい」との要望があった。これは、単なる進行状況だけでなく、どのプロジェクトのどのような詳細ステータスにあるかを把握したい意図と解釈される。

## 決定

`UnifiedTask` 型を拡張し、プロジェクト名と外部システム固有のステータス情報を保持・表示できるようにする。

### 1. 型の拡張（src/shared/types.ts）

`UnifiedTask` に以下のオプショナルフィールドを追加する：
- `projectName?: string`: 課題が属するプロジェクトの名前。
- `systemStatus?: { label: string; color?: string }`: 外部システムにおける元のステータス表示名。

### 2. データ取得の更新（pages/api/tasks/backlog.ts）

Backlog API から課題を取得する際、`project.name` を取得して `projectName` に設定する。また、マッピング前の `issue.status.name` を `systemStatus.label` に保持する。

### 3. UI の更新（pages/backlog.tsx）

- 課題タイトルの上または横に、プロジェクト名をバッジ形式で表示する。
- 統一ステータス（未対応/処理中など）の横に、必要に応じてシステム固有のステータス名を表示し、マッピングによる情報欠落を補う。

## 理由

- **複数プロジェクトの識別**: ユーザーが複数のプロジェクトを横断して作業している場合、プロジェクト名の表示は必須である。
- **情報密度の向上**: 汎用ステータスは背景色などの視覚的分類に使い、システムステータスは正確な状況把握に使うことで、利便性と柔軟性を両立させる。
- **拡張性**: 今後 GitHub Issues や他のソースを追加した際も、同様の枠組みでプロジェクト名やラベルを表示できる。

## トレードオフ・リスク

- **UI の複雑化**: 情報量が増えるため、モバイル表示などの狭い画面でレイアウトが崩れないよう配慮が必要。
- **ペイロードの微増**: `UnifiedTask[]` のデータサイズがわずかに増加するが、実用上の影響は軽微。

## 考慮した代替案

- **ラベルフィールドの活用**: 既存の `labels` 配列にプロジェクト名を入れる案もあったが、プロジェクト名は構造的に重要な情報であるため、明示的なフィールドとして扱う方が将来のフィルタリング等に役立つと判断した。
